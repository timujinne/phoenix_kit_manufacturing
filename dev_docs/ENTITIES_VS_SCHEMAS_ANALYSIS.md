# Анализ: данные производства как phoenix_kit_entities vs собственные схемы модуля

Вопрос владельца (2026-07-11): хранить карточки (станки, операции и т.п.) как записи phoenix_kit_entities — там уже есть включённые языки и создание новых полей; плюс нужен механизм «на каком языке вводим / на каком храним». Решение НЕ принимается — это разбор за/против.

## Что такое entities (по коду 0.2.x)
Универсальный CMS-движок: Entity (блюпринт: список полей из каталога 12 типов FieldTypes с валидацией) + EntityData (записи: data JSONB по ключам полей + metadata). Языковая поддержка и админ-UI полей — из коробки.

## ЗА entities
1. Готовые пользовательские поля: админ добавляет поле без миграции и без разработчика (наш field_template это уже даёт, но только в metadata и с 5 типами против 12).
2. Готовая мультиязычность значений на уровне движка.
3. Единый UI/UX управления справочниками, меньше собственного кода на новый справочник (причины брака как entity — почти бесплатно).
4. Меньше миграций в модуле.

## ПРОТИВ entities
1. **Потеря типизированной схемы**: станок перестаёт быть %Machine{} с колонками — всё в JSONB. Прощай: индексы по колонкам (location_uuid, status, manufacture_year-фильтры), FK-каскады (machine_operations!), Ecto-валидации, CHECK-констрейнты, честные запросы для планирования (v0.7: расчёты по нормам — JSONB-арифметика по EntityData будет мучительной и медленной).
2. **Связи M2M с атрибутами** (станок↔операция с переопределением нормы) в entities не выражаются — это реляционная связь с данными на ребре; пришлось бы городить обходные entity-связки без целостности.
3. **Интеграции между модулями** (склад v0.4: source kind production_order; PlacePicker; activity log resource_type) рассчитаны на конкретные схемы и uuid-контракты — через generic EntityData каждый контракт обрастает адаптером.
4. Зависимость от Hex-модуля entities (0.2.6, не наш форк) — своя скорость релизов, свои баги, чинить сложнее, чем свой код.
5. Производительность списков/фильтров/колонок: наш ColumnConfig-стек работает по полям структов; по EntityData — всё через data->>'key' без статических гарантий.
6. Уже сделано: паспорт+field_template+операции+брак реализованы и отревьюены (APPROVE) — миграция на entities = выбросить работающий типизированный слой ради генерализации, которая нужна лишь части полей.

## Рекомендация (гибрид, «ядро — схема, хвост — шаблон»)
- **Станки/операции/связи — остаются схемами модуля** (пп.1-3 ПРОТИВ решающие: планирование, M2M-нормы, межмодульные контракты).
- Гибкие поля уже покрыты field_template (наш аналог entities-полей, но типобезопасно ограниченный); при потребности расширяем каталог типов шаблона (file/rich_text) точечно.
- **Кандидаты на entities** (если захочется): будущие простые односущностные справочники без связей и без участия в расчётах (напр. «поставщики запчастей», «виды ТО» как заметки) — там entities дёшев и уместен.

## Мультиязычность ДАННЫХ (язык ввода/хранения)
Механика в платформе уже есть и использована в модуле: MultilangForm + data JSONB (primary-язык в колонке, остальные в data[locale]) — так уже работают machine_types и operations (name/description). Предложение по «указывать язык ввода»:
1. В формах с translatable-полями уже есть переключатель языка (LanguageSwitcher) — он и есть «язык ввода»; primary-язык системы = язык хранения канонической колонки.
2. Расширить на сами станки: name/description/notes станка сделать translatable тем же паттерном (сейчас plain) — маленькая задача, если нужно.
3. «Хранить на каком языке» отдельно от ввода — против: двусмысленность канонического значения; предлагаем правило платформы «канон = primary locale, остальное — переводы в data», как везде.

Итог: entities целиком — НЕ рекомендуем; мультиязычность данных станка — рекомендуем добавить существующим MultilangForm-паттерном (без entities), по команде владельца.
